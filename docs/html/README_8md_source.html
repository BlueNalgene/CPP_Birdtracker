3 # The LunAero Project
4 LunAero is a hardware and software project using OpenCV to automatically
5 track birds as they migrate in front of the moon. This repository contains
6 software used to track bird silhouettes from video produced by the LunAero
7 moontracking robot. For information on the hardware and software for the
8 robot, please see https://github.com/BlueNalgene/LunAero_C .
9 
10 Created by @BlueNalgene, working in the lab of @Eli-S-Bridge.

12 # Birdtracker_CPP vs. LunAero
13 
14 The Birdtracker_CPP repository is an updated version of the LunAero silhouette
15 tracking software. The previous implementation
16 (archived at https://github.com/BlueNalgene/LunAero) was a Python 3
17 script. This updated version is C++ code designed for Linux, and includes
18 many improvements, additional features, and is generally more mature code.

254 ## Behind the Curtain: What CPP_Birdtracker does
255 
256 This program executes birdtracking efficiently by splitting tasks
257 between CPU threads using the `fork()` command. Data which are shared
258 between the forks such as frame numbers and matrices, is shared using
259 memory mapping such that they can be accessed easily using pointers.
260 The `wait` command is used to sync across forks, and the script executes
261 code concurrently. When all frames of the video are completed, the data
262 are passed through post-processing and the program exits. A simplified
263 flowchart explains the process visually:
264 
265 ```
266  /-------\
267  | start |
268  \-------/
269  |
270  +---------+
271  | startup |
272  +---------+
273  |
274  +-------------+
275  | pre-process |
276  +-------------+
277  |
278  +--------+
279  | fork() |
280  +--------+
281  |
282  +------------+
283  | |
284  +-------+ +------+
285  | fetch | ->| sync | <------------------------\
286  +-->| next | | +------+ \
287  | | frame | | | \
288  | +-------+ | +--------+ \
289  | | | | fork() | \
290  | +-------+ | +--------+ \
291  | | cycle | | | \
292  | | frame | | +------------------------+ \
293  | | buffer| | | | \
294  | +-------+ | | +--------+ |
295  | | | +--------+ | cycle | |
296  | +-------+ / | Tier 1 | | frame | |
297  | | wait |/ +--------+ | buf 2 | |
298  | +-------+ | +--------+ |
299  | | +--------+ | |
300  | ^ | Tier 2 | +--------+ |
301  no | / \ +--------+ | Tier 3 | |
302  | / \ | +--------+ |
303  | / \ +-------+ | |while
304  |___/ video \ | wait/ | +--------+ |loop
305  \ done? / | sync | | Tier 4 | |
306  \ / +-------+ +--------+ |
307  \ / | | |
308  \ / \ +-------+ |
309  v \ | wait/ | |
310  yes | \ | sync | |
311  | \ +-------+ |
312  +--------------+ \ | /
313  | post-process | \ / /
314  +--------------+ \ / /
315  | \ / /
316  | \ / /
317  | \ / /
318  | \ / /
319  | +------+ /
320  | | kill |--------/
321  | | fork |
322  | +------+
323  /-----\
324  | end |
325  \-----/
326 ```
327 
328 Individual steps with more detail, but still simplified:
329 
330 - startup:
331  + Parses the input commands
332  + Redefines globals based on values from settings.cfg
333  + Creates directories/empty data files
334  + Instance and assign initial values to simple memory mapped regions
335 - pre-process
336  + Load the video
337  + Find the first frame where the moon is not touching the edge of the screen
338  + Record initial values
339  + Instance and assign initial values to complext memory mapped regions (frame buffers)
340  + Prepare for fork()
341 - FORK TWICE (three concurrent threads)
342  - fork0
343  + Check for exit command and sync status
344  + Coordinate sync values across forks
345  + Fetch the next frame
346  + Prep frame (color conversion and cropping)
347  + Store frame to the first buffer
348  + Wait for sync.
349  - fork1
350  + Check for exit command and sync status
351  + Run Tier 1 and Tier 2 calculations
352  + Wait for sync
353  - fork2
354  + Check for exit command and sync status
355  + Cycle first frame buffer with second frame buffer such that we have frame n and n-1
356  stored in the memory.
357  + Run Tier 3 and Tier 4 calculations (requires n and n-1 frames)
358  + Wait for sync
359 - Kill unused fork and continue loop
360 - If at end of the frames, kill all forks.
361 - post-process
362  + Create slideshow
363  + Concat data
364  + Crop slideshow
365 - Cleanup and Exit

367 ### Image Processing
368 
369 While there are a few image processing functions peppered through the code,
370 most of these are simple color conversions and cropping. The meat of the
371 image processing occurs within the "Tiers". This is where contours of
372 silhouettes crossing the moon are detected. They all operate on the same
373 frames, but some "Tiers" offer different capabilities. Each have strengths
374 and weaknesses.
375 
376 - Tier 1: Strict adaptive threshold. In perfect conditions, this finds
377 the majority of the large silhouettes. As the contours become less
378 clear or obvious, the image processing done on this Tier will begin to
379 miss some silhouettes.
380 - Tier 2: Loose adaptive threshold. In perfect conditions, this finds
381 most of the smaller silhouettes which may be missed by Tier 1. However,
382 it produces much more noise. Similar to Tier 1, as the silhouttes become
383 harder to see, they are less likely to be picked up here.
384 - Tier 3: Isotropic Laplacian filter. This is a very noisy filter which
385 compares differences in the Laplacian across two frames. The result is
386 blurred prior to detection, but much of the noise remains.
387 - Tier 4: UnCanny filter. This experimental filter attempts to perform
388 the steps used in the classic Canny filter backwards...across two frames.
389 The Sobel and power steps of the Canny filter occur on each frame (n and
390 n-1), then are added, rooted, and blurred. The result is passed through
391 an edge thinner and the silhouettes are detected.
392 
393 See the descriptions and code in the documentation
394 [here](https://bluenalgene.github.io/CPP_Birdtracker/html/frame__extraction_8cpp.html).
395 
396 Each Tier process has values which may be manipulated in `settings.cfg`
397 to customize the behavior during runtime.

419 ### The Data
420 
421 The data files in the `data` directory are as follows:
422 
423 | Filename | Description |
424 |--------------------|-------------------------------------------------------------|
425 | log.log | Debugging log of run |
426 | metadata.csv | Input video metadata |
427 | boxes.csv | Boundary values of largest contour in frame |
428 | ellipses.csv | Properties of the largest contour in frame |
429 | Tier1.csv | Tier 1 detected silhouettes |
430 | Tier2.csv | Tier 2 detected silhouettes |
431 | Tier3.csv | Tier 3 detected silhouettes |
432 | Tier4.csv | Tier 4 detected silhouettes |
433 | mixed_tiers.csv | All tier data mixed into a single file |
434 | offscreen_moon.csv | Number of pixels where the moon is touching the screen edge |
435 
436 - log.log - If the boolean toggle `DEBUG_COUT` is set to `true` in
437 settings.cfg, this log file will be created. Debugging messages are
438 written to this file over the course of the run. Note, that this does
439 not include any STDOUT or STDERR messages. If there is an error
440 message written to the terminal, for example, you must fetch that another
441 way.
442 - metadata.csv - This file lists properties of the input video.
443 Currently, that just includes the path to the input video.
444 - boxes.csv - This comma separated values (csv) file is mostly used for
445 internal calculations. It is retained for diagnostic conveneince. The
446 contents are values calculated which describe the location and size of
447 the boundary points of the largest contour on each frame (presumably,
448 this is the moon).
449 - ellipses.csv - This csv file lists properties of the largest contour
450 on the screen more relevant to lunar calculations. Includes information
451 such as the major and minor axis of the contour, area, and edges
452 which touch the side of the screen.
453 - Tier1.csv - This csv lists all of the silhouettes detected using the
454 stricter version of the `adaptiveThreshold` method. See
455 [tier_one()](https://bluenalgene.github.io/CPP_Birdtracker/html/frame__extraction_8cpp.html#a4f9b7b156d48ad3f147ff2b4edc01509)
456 - Tier2.csv - This csv lists all of the silhouettes detected using the
457 more lenient version of the `adaptiveThreshold` method.
458 - Tier3.csv - This csv lists all of the silhouettes detected using
459 isotropic Laplacian matching between frames.
460 - Tier4.csv - This csv lists all of the silhouettes detected using
461 a new filter we are calling "UnCanny", which is a reverse operation
462 of the Canny filter between two frames.
463 - mixed_tiers.csv - This csv is a convenience file which inclues
464 everything from Tier*.csv in frame order. An additional column
465 indicates which Tier method the line was generated from. Only created
466 if `CONCAT_TIERS` is set to `true`.
467 - offscreen_moon.csv - This csv is a convenience file which edits down
468 ellipses.csv to only report frames with non-zero values in one or more
469 of the screen edge columns. Only created if `SIMP_ELL` is set to
470 `true`.

472 ### The Video
473 
474 If the `OUTPUT_FRAMES` value in settings is set to `true`, the Birdtracker
475 will output every cropped frame to the `frames` directory as a `png` file
476 with the naming format based on the frame number. These frames will remain
477 in the directory unless the setting `EMPTY_FRAMES` is set to `true`.
478 
479 If the program is outputting frames, and `GEN_SLIDESHOW` is set to `true`,
480 the frames will be contatenated into a slideshow saved as `output.mp4`.
481 
482 If the program generates a slideshow, and `TIGHT_CROP` is set to `true`,
483 the output video will be modified at the end of the run to crop each frame
484 to a smaller size. The size of these frames is based on the extreme
485 edges of the moon as reported in `boxes.csv` + 10 pixels. The cropped
486 video is saved as `output.mp4`.
